1. Короткий опис продукту

SkinSwap — це безпечна вебплатформа для обміну ігровими предметами (сскінами) між користувачами ігор CS та Roblox. Проблема полягає у високому ризику шахрайства при прямих обмінах поза офіційними майданчиками. Система забезпечує цінність через впровадження механізму Escrow (утримання предметів сервісом-посередником), що гарантує виконання умов угоди обома сторонами.

2. Стейкхолдери та їхні інтереси
Вимоги сформовані з урахуванням потреб основних зацікавлених сторін:

Геймер-трейдер: хоче проводити безпечні обміни без ризику втрати активів. Успіхом для нього є отримання бажаного предмета за справедливою ціною.

Адміністратор платформи: прагне автоматизувати процеси перевірки угод. Успіх — мінімальна кількість конфліктних тікетів у підтримці.

Розробник: зацікавлений у стабільності інтеграції з API ігрових платформ (Steam, Roblox). Успіх — надійна робота сервісу без технічних збоїв API.

Служба безпеки: очікує повного логування дій для запобігання відмиванню грошей або передачі крадених предметів.

3. Функціональні вимоги (FR)
Функціональні вимоги описують поведінку системи та дії, які вона повинна виконувати:
FR-01: Система повинна забезпечувати авторизацію користувачів через Steam OpenID та OAuth Roblox.

FR-02: Система повинна автоматично завантажувати та відображати предмети з публічного інвентарю користувача.

FR-03: Система повинна надавати функціонал створення унікального посилання на пропозицію обміну.

FR-04: Система повинна блокувати передачу предмета («заморожувати» на боті-посереднику) до моменту взаємного підтвердження.

FR-05: Система повинна автоматично оцінювати ринкову вартість предметів на основі даних сторонніх маркетплейсів.

FR-06: Система повинна вести історію транзакцій користувача з відображенням статусів (Pending, Completed, Cancelled).

4. Нефункціональні вимоги (NFR)
Описують якість і обмеження системи:
NFR-01 (Продуктивність): Час завантаження інвентарю для 95% запитів не повинен перевищувати 3 секунди.
+1

NFR-02 (Безпека): Усі транзакційні дані та сесії повинні бути захищені протоколом HTTPS з TLS 1.3.

NFR-03 (Доступність): Система повинна бути доступною 99.9% часу на місяць (Uptime).

NFR-04 (Масштабованість): Архітектура повинна підтримувати обробку до 1000 одночасних обмінів без деградації швидкості.

NFR-05 (Зручність): Інтерфейс платформи має бути адаптивним (Responsive) для коректного відображення на мобільних пристроях.

NFR-06 (Надійність): У разі недоступності Steam API, система повинна виводити інформаційне повідомлення користувачу протягом < 1 секунди.

7. Ризики та способи їх зменшення (R)

R-01: Тимчасове блокування ботів-посередників з боку Steam. Зменшення: Використання декількох ботів та імітація поведінки реального користувача під час обміну.

R-02: Висока волатильність цін на скіни. Зменшення: Автоматичне оновлення цін через API маркетплейсів кожні 5 хвилин.

R-03: Фішинг та підміна посилань. Зменшення: Впровадження унікального «секретного коду» для кожної транзакції, який відображається лише в інтерфейсі сервісу.
